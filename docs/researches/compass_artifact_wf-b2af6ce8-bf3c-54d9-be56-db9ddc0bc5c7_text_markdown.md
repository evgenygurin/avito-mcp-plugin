# Парсинг Avito на Python (2025–2026): антибот, инструменты, прокси, архитектура, право

## TL;DR
- **Avito защищён Qrator/CURATOR Antibot + фаервол-капча (hCaptcha Enterprise / GeeTest).** «Голый» HTTP-клиент получает `403` «Доступ ограничен: проблема с IP». TLS/JA3-импersonation (curl_cffi) снимает транспортный слой, но НЕ проходит Qrator JS-challenge (кука `qrator_jsid`) — на практике нужен либо реальный браузер для генерации кук, либо чистые RU-резидентные/мобильные прокси, чтобы капча вообще не всплывала.
- **Рабочая production-схема — гибрид:** реальный браузер (Playwright/patchright) генерирует куки → куки+прокси передаются в async HTTP-клиент (curl_cffi/httpx) для быстрых запросов к внутреннему JSON-API (`m.avito.ru/api/*`). Именно так устроен один из самых живых open-source парсеров под Avito (Python 3.11+, поддержка до июня 2026).
- **Правовой риск в РФ реален, но не абсолютен:** ст. 1334 ГК (БД >10 000 элементов, компенсация до 5 млн ₽), 152-ФЗ (телефоны/имена — ПДн), ст. 272 УК (обход техсредств защиты). При этом Avito ПРОИГРАЛ разбирательство в ФАС против Auto.ru по копированию объявлений. Ключевое: не копировать существенную часть БД, не собирать ПДн для перепродажи, не ронять сервер.

## Key Findings

1. **Антибот Avito = Qrator Labs (в РФ ныне CURATOR) Antibot.** Механизм: JS-fingerprinting, JA3/JA4-фильтрация, поведенческий анализ, блок скрапера с первого запроса, кука `qrator_jsid` как маркер прошедшего проверку. Фаервол-капча Avito использует hCaptcha Enterprise + GeeTest, эндпоинт `/web/5/firewallCaptcha/get` с параметром `rqdata`.
2. **curl_cffi обязателен, но недостаточен.** Он побеждает TLS/JA3/HTTP2-fingerprint, но не исполняет JS → не решает Qrator-challenge. Официальная FAQ curl_cffi это подтверждает: «For the most basic ones, TLS fingerprints alone maybe enough, but for higher levels, you may need to find a better proxy IP provider and use browser automation tools like playwright».
3. **Headless детектится агрессивно.** Практики сообщают: headless-Chromium почти всегда получает капчу/пустую заглушку; headful (или Xvfb) + RU-резидентные прокси работает стабильнее.
4. **Официальный Avito API существует, но только для своих объявлений** (продавцы/автосалоны/реклама), а не для массового сбора чужих. OAuth2 client_credentials, base `https://api.avito.ru/`, rate-limit в заголовках `X-RateLimit-*`, у рекламного API — модель «баллов».
5. **Прокси: только RU. Мобильные > резидентные > датацентр.** Датацентр банится мгновенно. Порядок цен: мобильные ~2500–5000 ₽/мес, резидентные ~290 ₽/ГБ.
6. **По-настоящему живой репозиторий на GitHub практически один.** Большинство прочих (2020–2022) мертвы.

## Details

### 1. Актуальное состояние антибот-защиты (2025–2026)

**Провайдер защиты — Qrator Labs.** В декабре 2024 российское юрлицо (ООО «Эйч-Эль-Эль», гендиректор Дмитрий Ткачёв, конференция «Код ИБ. Итоги 2024», 5 дек. 2024) объявило о смене названия российского бизнеса с Qrator Labs на **Curator** (товарный знак зарегистрирован в Роспатенте 15 августа 2024 г., домен curator.pro); зарубежный бизнес под руководством Александра Лямина остался Qrator Labs. Продукт Qrator.AntiBot / CURATOR.ANTIBOT входит в платформу фильтрации трафика. Подтверждённые сообществом механизмы применительно к Avito:

- **JS-fingerprinting.** Qrator проверяет цифровой отпечаток браузера JS-кодом. «Браузеры с плохим цифровым отпечатком или приложения без поддержки JS (включая парсеры без использования браузеров) получат настраиваемую страницу блокировки» (документация CURATOR). Легитимный юзер кратко видит check-page (пустую/401), затем получает контент и куку `qrator_jsid`.
- **JA3/JA4 TLS-фильтрация** и анализ цепочек HTTP-запросов — заявлено в описании российских антибот-решений (Qrator/CURATOR, Servicepipe, StormWall).
- **Поведенческий анализ** — «внезапная капча после нескольких действий» (анализ мыши, скролла, кликов).
- **Rate limiting и блок по IP/подсети.** Ответ `403` / страница «Доступ ограничен: проблема с IP» для датацентр-IP и иностранных IP. `429` при частых запросах.
- **Требование авторизации для чувствительных действий.** Телефон продавца скрыт за XHR; для неавторизованного клик по «Показать телефон» открывает окно логина.
- **Фаервол-капча.** Тип — hCaptcha Enterprise + GeeTest. Детект через перехват эндпоинта `/web/5/firewallCaptcha/get` (`[N]` — версия), передачу `rqdata`. В headless hCaptcha Enterprise часто не дорисовывается — «решать просто нечего» (dmitriy1996-cmd, Habr 1053772). Именно эти системы (hCaptcha, GeeTest, Turnstile, DataDome) перечисляются в FAQ CapSolver/curl_cffi как JS-challenge, которые чистый TLS-клиент не проходит.

**Уровни защиты (сводно):**

| Уровень | Механизм | Признак |
|---|---|---|
| Сетевой | Блок IP, rate limiting | 403/429, бан IP, «проблема с IP» |
| Транспортный | TLS/JA3/JA4 fingerprint | Блок до HTTP-ответа |
| Прикладной | Qrator JS-challenge, hCaptcha Enterprise/GeeTest | Пустая страница / капча |
| Поведенческий | Мышь, скролл, паттерны | Внезапная капча на 30–50+ запросе |

### 2. Подходы к парсингу

#### 2.1 HTTP-клиенты

**curl_cffi** — де-факто стандарт для обхода TLS-fingerprint в Python. Умеет impersonate Chrome/Safari (JA3/TLS + Akamai HTTP2), async с ротацией прокси на запрос, HTTP/2 и HTTP/3. Ключевое ограничение (официальная FAQ и ScrapingBee/ProxyHat, 2026): **не исполняет JavaScript**, поэтому JS-challenge и Turnstile/hCaptcha он не проходит. Против Avito: снимает транспортный слой, но Qrator JS-challenge остаётся.

```python
from curl_cffi import AsyncSession

async def fetch(url, proxy, cookies):
    async with AsyncSession(impersonate="chrome", proxies={"https": proxy}) as s:
        r = await s.get(url, cookies=cookies, headers={
            "Accept-Language": "ru-RU,ru;q=0.9",
            "Referer": "https://www.avito.ru/",
        })
        return r
```

Совет из практики: использовать `impersonate="chrome"` (не пиновать `chrome124`), ставить рандомные задержки 1–3с, полный набор заголовков, одна сессия на один IP.

**httpx / aiohttp** — не спуфят TLS, в чистом виде против Avito бесполезны, но httpx удобен как API-обёртка (в подобных живых парсерах Avito curl_cffi нередко подключают транспортом к httpx). **tls-client, primp** — альтернативы curl_cffi; primp новее и быстрее, tls-client стабилен, но обновляется реже.

#### 2.2 Headless-браузеры и антидетект

По независимым бенчмаркам 2026 (techinz/browsers-benchmark, ianlpaterson.com) против антибот-систем:

| Инструмент | База | Стелс против сильных антиботов | Против Avito (практика) |
|---|---|---|---|
| **Camoufox** | Firefox (C-level spoofing) | Топ (0% детекта в CreepJS) | Работает, но Firefox TLS иногда флагается |
| **nodriver/zendriver** | CDP напрямую, без WebDriver | Очень высокий (0 блоков в бенче) | Хорош, наследник undetected-chromedriver |
| **patchright** | Playwright fork (фикс CDP-утечек) | Высокий, drop-in | Рабочий вариант |
| **SeleniumBase UC** | undetected-chromedriver | Высокий, авто-CAPTCHA | Проверенный RU-выбор |
| **playwright-stealth** | Playwright + плагин | Средний | Базовый, часто ловит капчу |

Практический вывод по Avito: **headless детектится**, нужен headful или Xvfb. Camoufox и nodriver — сильнейшие open-source; patchright — самый простой drop-in для существующего Playwright-кода. Все они требуют регулярного патчинга под новые версии браузеров.

#### 2.3 Гибридная схема (рекомендуемая)

Браузер (Playwright/patchright/Camoufox) с RU-резидентным прокси → проходит Qrator JS-challenge → забирает куки (`qrator_jsid` и пр.) → куки+прокси передаются в curl_cffi/httpx для быстрых запросов к JSON-API. Именно так устроена типичная реализация живых open-source парсеров Avito: отдельный модуль на Playwright для кук + curl_cffi-бэкенд под httpx для запросов; `requests` — для уведомлений. Браузер можно отключить конфиг-флагом для серверов с малым RAM.

#### 2.4 Мобильный API и JSON в HTML

- **Внутренний API мобильной версии:** `https://m.avito.ru/api/9/items`, `/api/11/items`, `/api/15/items/{id}`, телефон — `/api/1/items/{id}/phone`. Требуется параметр `key` (зашит во фронт; «гуглящийся» статичный ключ `af0deccbgcgidddjgnvljitntccdduijhdinfgjgfjir` встречается на форумах; меняется при обновлениях фронта). Способ в десятки раз быстрее браузера, но нестабилен — эндпоинты/параметры меняются без предупреждения.
- **Данные в HTML:** контент рендерится JS, но начальное состояние часто лежит в JSON внутри страницы (`__initialData__`/Next.js-подобный state) и в JSON-LD (использует, напр., EduardBardackiy/Avito-Parser для картинок). `data-marker`-атрибуты стабильнее CSS-классов (классы меняются раз в 1–2 недели, `data-marker` — раз в 1–3 месяца).

#### 2.5 Официальный Avito API

Существует и документирован (Swagger), но это API **для управления своими объявлениями и рекламой**, не для массового сбора чужих:

- **Авторизация:** OAuth2, `grant_type=client_credentials` (и Authorization Code для части методов), `GET https://api.avito.ru/token/`. Client ID и secret выдаются бесплатно в кабинете.
- **Scopes:** `items:info`, `items:apply_vas`, `autoload:reports`, `job:*`, `messenger:read/write`, `autoteka:*` и др.
- **Методы:** информация об объявлениях, статистика, автозагрузка, мессенджер, Автотека, категории Недвижимость/Работа.
- **Лимиты:** rate-limit в заголовках `X-RateLimit-Limit` / `X-RateLimit-Remaining`. У API Авито Рекламы отдельная модель — «баллы», восстанавливаются еженедельно (понедельник 00:00 UTC), плюс независимый rate-limit на каждый метод; ключ API создаёт роль Администратора кабинета, есть песочница.
- **Отличие от парсинга:** легально, стабильно, но только свои данные. Массовый сбор чужих объявлений официальным API невозможен.

Python-обёртки: `DUB1401/AvitoAPI` (клиентское API, в разработке), `covox/avito_api` (PHP-swagger). Есть также сторонние платные «API к чужой базе» (ads-api.ru — пауза ≥5с между запросами, rest-app.net) — де-факто чужой парсинг как услуга; это НЕ официальный Avito.

### 3. Прокси-инфраструктура

- **Только RU-IP.** Иностранные и датацентр-IP → фаервол-заглушка сразу.
- **Иерархия устойчивости:** мобильные > резидентные > датацентр. За одним мобильным IP — тысячи реальных абонентов, Avito не может забанить без ущерба.
- **Стоимость (порядок величины, RU-рынок):** мобильные индивидуальные ~2500–5000 ₽/мес (mobileproxy.space ~2500 ₽/мес; proxys.io персональные 3999 ₽/мес, премиальные 5249 ₽/мес); резидентные ~290 ₽/ГБ (ProxySeller); статический серверный ~120–150 ₽/мес (бесполезен для Avito).
- **В сообществе часто рекомендуют mobileproxy.space** для RU-мобильных прокси (скидка до 20%, страна Россия) — плюс сервис готовых кук spfa.ru.
- **Ротация:** sticky-сессия на поток (фиксированный выходной IP на время обработки объявления), смена IP+фингерпринта только на ретрае/бане. Ротация на каждый запрос провоцирует антифрод. Смена контекста браузера каждые 15–20 запросов (сброс cookies/localStorage/IP).
- **Паузы:** 2–6с между действиями, длинная пауза (30–60с) каждые ~20 объявлений. Клики «Показать телефон» — ≤5–10/мин.

### 5. Архитектура production-решения (под стек: FastAPI, async SQLAlchemy, PostgreSQL, Redis, Celery/Kafka, Docker/K8s)

**Топология:**
- **Слой добычи кук (browser pool):** отдельный Deployment с Playwright/patchright под Xvfb, headful, 1 браузер = 1 RU-прокси. Складывает валидные куки (`qrator_jsid` + прочие) в Redis с TTL (~10–12ч по опыту cookie-сервисов).
- **Слой запросов (async workers):** curl_cffi (`impersonate="chrome"`, `http2=False` для стабильности — распространённая практика в живых парсерах Avito) + asyncio; берёт куки+sticky-прокси из Redis, ходит в `m.avito.ru/api/*` или парсит HTML.
- **Очереди:** Celery для периодических/ретрай-задач; Kafka как шина событий (topic `listings.raw` → парсинг → `listings.parsed` → загрузка в БД). Redis — rate-limit (token bucket на прокси/домен) и дедуп-кэш.
- **Мониторинг банов:** счётчик 403/429/капч по прокси в Redis; при превышении порога — прокси в карантин, алерт (Prometheus + Grafana).

**Схема БД (PostgreSQL), инкрементальный парсинг и история цен:**

```sql
CREATE TABLE listings (
    id            BIGINT PRIMARY KEY,          -- avito item id
    url           TEXT NOT NULL,
    title         TEXT,
    category      TEXT,
    region        TEXT,
    seller_name   TEXT,
    seller_type   TEXT,
    price         NUMERIC,
    address       TEXT,
    lat           DOUBLE PRECISION,
    lng           DOUBLE PRECISION,
    params        JSONB,
    first_seen    TIMESTAMPTZ DEFAULT now(),
    last_seen     TIMESTAMPTZ DEFAULT now(),
    is_active     BOOLEAN DEFAULT true,
    content_hash  TEXT                          -- хэш значимых полей для детекта изменений
);

CREATE TABLE price_history (
    listing_id  BIGINT REFERENCES listings(id),
    price       NUMERIC NOT NULL,
    changed_at  TIMESTAMPTZ DEFAULT now(),
    PRIMARY KEY (listing_id, changed_at)
);
CREATE INDEX idx_listings_active_seen ON listings(is_active, last_seen);
```

**Дедупликация + отслеживание изменения цены (async SQLAlchemy, UPSERT):**

```python
from sqlalchemy.dialects.postgresql import insert
from sqlalchemy import func

async def upsert_listing(session, item: dict):
    stmt = insert(Listing).values(**item)
    stmt = stmt.on_conflict_do_update(
        index_elements=[Listing.id],
        set_={
            "price": stmt.excluded.price,
            "last_seen": func.now(),
            "is_active": True,
            "content_hash": stmt.excluded.content_hash,
        },
        where=(Listing.content_hash != stmt.excluded.content_hash),
    )
    await session.execute(stmt)
    # если цена изменилась — отдельным запросом пишем в price_history
```

**Retry/backoff с ротацией:**

```python
import asyncio, random

async def fetch_with_retry(url, get_proxy, get_cookies, tries=3):
    for attempt in range(tries):
        proxy, cookies = get_proxy(), get_cookies()
        r = await fetch(url, proxy, cookies)
        if r.status_code == 200 and "firewallCaptcha" not in r.text:
            return r
        mark_bad(proxy)                       # карантин + новые куки
        await asyncio.sleep(2 ** attempt + random.random())
    raise BlockedError(url)
```

**Деплой:** browser-pool и http-workers — раздельные Deployments (у браузеров высокие CPU/RAM-лимиты, у HTTP-воркеров — низкие); прокси-креды в Secrets; HPA по глубине очереди Kafka/Celery. Xvfb в контейнере браузера (headless детектится).

### 7. Правовые аспекты (РФ)

- **Смежное право на БД (ст. 1334, 1335.1 ГК).** Сайт с ≥10 000 элементов презюмируется охраняемой БД. Нарушение — извлечение всей/существенной части ИЛИ систематическое извлечение несущественных частей во вред нормальному использованию. Компенсация 10 000 – 5 000 000 ₽ (ст. 1301). Avito прямо декларирует защиту БД в пользовательском соглашении (водяные знаки на фото).
- **Ключевой прецедент — Avito против Auto.ru.** По данным «Коммерсанта» и «Известий», руководитель категории «Автомобили» Avito Кирилл Вотяков заявлял о копировании масштаба «20% объявлений о продаже автомобилей от частных продавцов на "Авто.ру", или 70 тыс. объявлений в месяц» (ноябрь 2018). **ФАС решением от 15 апреля 2019 г. отказала в возбуждении дела против ООО «Яндекс.Вертикали» (Авто.ру), «не найдя признаков нарушения закона о защите конкуренции»:** контакт с пользователями — обычная практика; публикация на одной площадке не мешает другой; информация об авто принадлежит продавцам, а не Avito. Вывод: доказать нарушение через парсинг в РФ Avito сложно.
- **ВКонтакте против Double Data (А40-18827/2017).** Иск подан в январе 2017; Double Data оценивала кредитоспособность заёмщиков по заказу банков. Спор длился ~6 лет: 23 сентября 2022 г. Суд по интеллектуальным правам (кассация) утвердил мировое соглашение сторон. Президиум СИП ранее (Обзор от 18.11.2021 № СП-21/26) подтвердил, что владелец соцсети защищён смежным правом изготовителя БД. Систематический коммерческий скрапинг суды склонны считать нарушением, даже если данные публичны.
- **HeadHunter против «Стафори».** Суд отказал HH (не доказан доступ к закрытой части); параллельно ФАС признала блокировку парсинга ограничением конкуренции. Тотальный запрет парсинга в соглашении сам может быть антимонопольным нарушением.
- **152-ФЗ (персональные данные).** Телефоны, имена, email — ПДн. Роскомнадзор и Арбитражный суд Москвы: открытость профиля ≠ согласие на обработку; «соцсети не являются источником общедоступных ПДн». Сбор контактов для обзвона/перепродажи — зона высокого риска.
- **Ст. 272 УК (неправомерный доступ).** Применяется при обходе техсредств защиты (капча, блокировки, лимиты через прокси) ИЛИ если парсинг повлёк уничтожение/блокирование/модификацию/копирование охраняемой информации. Крупный ущерб — свыше 1 млн ₽; ч.1 — до 2 лет, при отягчающих — до 7 лет. Нюанс из практики: информация в открытом доступе без спецрежима защиты (ценники, открытые объявления) обычно НЕ признаётся «охраняемой законом компьютерной информацией» — но обход именно капчи/антибота смещает квалификацию в сторону ст. 272. Возможна и ст. 273 (если скрипт квалифицируют как вредоносную программу для нейтрализации защиты).
- **robots.txt.** Соблюдение — «сетевой этикет», не закон, но его игнор используется как довод против парсера.

**Что реально грозит:** гражданский иск (компенсация до 5 млн ₽ за БД, моральный вред за ПДн, убытки за перегрузку сервера), предписание удалить данные; при обходе антибота и сборе ПДн для коммерции — риск ст. 272/273 УК. Наименьший риск: только фактические данные (цены, характеристики), без существенной доли БД, без ПДн на перепродажу, без перегрузки сервера.

## Recommendations

1. **Старт (PoC, ≤50 объявлений):** curl_cffi `impersonate="chrome"` + 1 RU-мобильный прокси, headful Playwright только для добычи кук. Проверить структуру `m.avito.ru/api/*` через DevTools. Порог смены стратегии: если >20% ответов — капча/403.
2. **MVP (сотни-тысячи/день):** гибрид browser-pool (Playwright/patchright под Xvfb) → Redis (куки) → async curl_cffi workers. Sticky-сессии, паузы 2–6с, RU-мобильные прокси (пул 5–10). Ориентир пропускной способности из практики: ~500–1500 объявлений/день на 1 прокси, до 5000–10000 на пул 5–10. Порог: рост доли банов >10% → добавить прокси / снизить темп.
3. **Production:** вынести дедуп/историю цен в PostgreSQL (UPSERT + content_hash), очереди на Celery+Kafka, rate-limit и карантин прокси в Redis, мониторинг банов в Prometheus. Инкрементальный парсинг по `last_seen`/`content_hash`, а не полный обход.
4. **Если данные нужны легально/стабильно:** для своих объявлений — официальный Avito API (OAuth2). Для разовой аналитики — Apify-актор под Avito. Порог перехода на сервис: когда стоимость поддержки антибот-обхода (инженеро-часы) превышает $300–500/мес.
5. **Юридически:** собирать только фактические поля (цена, площадь, характеристики), не телефоны/имена для перепродажи; не копировать существенную долю БД; не обходить капчу «в лоб» (провоцирует ст. 272) — вместо решения капчи избегать её ротацией чистых IP; соблюдать robots.txt и не ронять сервер.

## Caveats
- **Точные метрики Avito (реальные лимиты запросов, конкретные пороги бана, версии Qrator) публично не раскрываются** — цифры пропускной способности взяты из отчётов практиков (Habr, codesrc.ru) и являются оценками, не официальными данными.
- **Внутренний `key` мобильного API и эндпоинты меняются без предупреждения** — код-примеры иллюстративны, требуют актуализации через DevTools.
- **Зависимость curl_cffi в подобных живых парсерах подтверждается лишь косвенно** (сигнатура ошибки `curl: (28)`, упоминания http2/httpx в issue-трекерах и changelog'ах), точный pin версии в requirements.txt верифицировать не удалось. Оценки популярности таких репозиториев (звёзды, форки) быстро устаревают и расходятся между источниками.
- **curl_cffi против Avito в одиночку, скорее всего, недостаточен** — консенсус сообщества и FAQ curl_cffi: против Qrator JS-challenge нужен реальный браузер либо готовые куки.
- **Antidetect-браузеры — движущаяся мишень:** Camoufox пережил ~год без обновлений (возобновлён в конце 2025), stealth-инструменты требуют патчинга под каждую версию браузера.
- **Правоприменение по ст. 272 к «чистому» парсингу открытых данных неоднородно** — прямых обвинительных приговоров именно за скрапинг Avito в открытых источниках не найдено; риск возрастает при обходе капчи и сборе ПДн.
